# SQL関連関数の完全互換再実装 検討レポート

> 作成: 2026-07-03（32bit フォーク / FMP11 対象）
> 提案: 「SQL関連関数（`BE_FileMakerSQL` 系）を、API と入出力を完全互換のまま新規に生成（再実装）するのが、問題解決として早いか」を検討する
> 関連資料: `BUGREPORT-fmp11-idle-executefilesql.md`、メモリ `be-plugin-audit-fixes` / `be-plugin-shellexec-rewrite` / `be-plugin-xml-msvc-static`

---

## 1. 結論（先に要点）

**再実装は近道ではない。むしろ唯一成立しない選択肢である。**

理由は単純で、**SQL の実行エンジンはプラグイン側ではなく FileMaker ホスト側にある**からだ。`BE_FileMakerSQL` の実体は `fmx::ExprEnv::ExecuteFileSQL(TextResult)` の薄いラッパーであり、「書き直せるコード」がほぼ存在しない。問題（FMP11 の idle クラッシュ）はホスト側 API の制約であって、ラッパーをどう書き直しても同じ API を呼ぶ限り同じ結果になる。

さらに重要な点として、**この問題は既に解決済み**である（コミット `9031d64b` の extnVersion ゲート + 呼び出し元 ExprEnv でのドレイン。FMP19 実機で idle DDL リプレイの安全性も検証済み）。残っている実害は「FMP11 では DDL が実行されない（クラッシュの代わりに機能制限）」の一点だけであり、これは再実装では解消できない（§4）。

**推奨: 現状維持（ゲート方式）。追加でやる価値があるのは §5 の小実験1件のみ。**

---

## 2. なぜ「書き直しで解決」の成功パターンが当てはまらないか

このフォークでは「API・入出力互換のまま実装を入れ替える」戦略が2回成功している。だが両者と SQL では**ロジックの所在**が決定的に違う。

| 対象 | 壊れていた層 | ロジックの所在 | 書き直し可否 |
|---|---|---|---|
| シェル実行（BEShellExec） | BE のラッパー層 + FMWrapper 依存 | **プラグイン内**（CreateProcessW / パイプ / エンコーディング） | ○ ゼロ再設計で成功 |
| XML（BE_XPath 等） | ラッパー層 + mingw ビルドの libxml2 | **プラグイン内**（静的リンクした libxml2/libxslt） | ○ MSVC /MT 静的化で成功 |
| **SQL（BE_FileMakerSQL）** | ホスト側（FMP11 の idle 再入） | **FileMaker ホスト内**（fmDatabase の SQL エンジン） | **× 入れ替える実装が存在しない** |

シェルと XML は「プラグインが自前で持っているエンジン」を差し替えた。SQL のエンジンは FileMaker の中にあり、プラグインからは `ExprEnv` 経由でしか触れない。つまり `BESQLCommand.cpp`（約290行）を何度書き直しても、最後の行は必ず `environment.ExecuteFileSQL(...)` になる。

## 3. 現状アーキテクチャと問題の正確な切り分け

### 3.1 SQL 関連のコード構成

- `BE_FileMakerSQL`（`BEPluginFunctions.cpp:4473`）— 引数解析（式・区切り文字・ファイル名・text/binary・ファイル出力）だけ。実行は `BESQLCommand` に委譲
- `BESQLCommand::execute(env, ...)`（`BESQLCommand.cpp:101`）— SELECT/DML は同期実行。DDL（ALTER/CREATE/DROP 前方一致）は `g_ddl_command` にキューして idle でリプレイ（上流由来の設計）
- `BESQLCommand::execute()` 引数なし版（`:92`）— `FMX_SetToCurrentEnv` で環境を取得。**FMP11 で idle から呼ぶとこれが 0xc0000409 で即死**（BUGREPORT 参照）
- 背景タスクの SQL キュー（M-28）— 同ファイル。FMP11 では idle でなく次のプラグイン関数呼び出しの ExprEnv でドレインする方式に修正済み

### 3.2 問題の本質

クラッシュの犯人は「BE のコードの品質」ではなく **「FMP11 において kFMXT_Idle コールバックから FM API の SQL 実行系を呼ぶこと」自体**。同じ SQL・同じコードが通常の外部関数呼び出しの `ExprEnv&` では正常に動くことを実機で確認済み。FMP19 では同一コードパスがクラッシュしないことも確認済み。

つまり「入出力完全互換の新実装」を作っても、実行手段が FMX API である限り FMP11 の制約をそのまま引き継ぐ。書き直しで変わるものが何もない。

## 4. 「FMX を使わない」完全互換代替経路は存在するか

エンジンがホスト内にある以上、FMX 以外の経路は「外から FileMaker に SQL を投げる」方式しかない。候補を潰しておく。

### 4.1 xDBC（ODBC/JDBC 共有）経由 — 互換不能

FMP11 はローカル ODBC 共有を持つので技術的には接続できるが、「完全互換」の要件を複数の点で満たせない。

- **認証**: `ExecuteFileSQL` は現在のセッション権限で動く。ODBC はアカウント名＋パスワードが必須で、プラグインはそれを知らない。引数追加＝入力互換の放棄
- **セットアップ**: 各ファイルで ODBC/JDBC 共有を有効化＋クライアント側に FileMaker ODBC ドライバのインストールが必要。「プラグインを入れれば動く」が崩れる
- **出力互換**: コンテナ（`BE_FileMakerSQL` の text_result=false でのバイナリ返却）、数値・日付の文字列化規則が xDBC では別物になる
- **信頼性**: FMP11 の 32bit ODBC ドライバ＋Windows 11 という組み合わせ自体が追加の互換性リスク

→ 「動く別関数」は作れても「完全互換の BE_FileMakerSQL」にはならない。労力は BEShellExec 再設計の数倍で、得られるのは劣化互換。

### 4.2 プラグイン内に SQL エンジンを持つ — 論外

データが .fmp7 ファイル（ホストのメモリ/ロック管理下）にある。ファイルフォーマットを直接読む実装は非公開仕様のリバースエンジニアリングであり、書き込み（DDL/DML）は破損リスクが非現実的に高い。検討対象外。

### 4.3 FMX_StartScript でスクリプト経由実行 — 同じ穴

idle から `FMX_StartScript` でスクリプトを起動し、スクリプト内の計算式でプラグイン関数を呼び戻す案。しかし FMP11 の `FMX_StartScript` が idle から安全である保証がなく（同族の再入問題）、しかも結局スクリプト側の評価コンテキスト＝通常の ExprEnv に戻るだけで、現行の「次の関数呼び出しでドレイン」方式と等価。追加の複雑さに見合う利得がない。

## 5. 残ギャップ（FMP11 の DDL）と現実的な選択肢

現状の唯一の機能欠落: FMP11 では DDL が `g_ddl_command` にキューされたまま実行されない（ドキュメント化済みの仕様）。

上流が DDL を idle 送りにするのは「計算式の評価中は DDL が実行できない」という FM API の制約が前提。だが **その制約が FMP11 でどの程度厳密かは未検証**。そこで:

- **小実験（推奨・数十分）**: FMP11 実機で、通常の関数呼び出しの `ExprEnv&` から DDL を**同期実行**してみる（`waiting` フラグを立てて再キューを回避するだけの数行パッチ）。
  - 成功（またはエラーコードが返るだけ）なら → FMP11 分岐では「DDL も同期実行」に切り替えて機能欠落が消える。エラーが返る場合でも「クラッシュせず 8xxx エラー」なら、`g_last_ddl_error` に載せてユーザーに見せられる分だけ現状より良い
  - ハングやクラッシュなら → 現状の「FMP11 では DDL 不可」を最終仕様として確定
- **やらないこと**: xDBC ブリッジ・SQL エンジン内蔵・idle 機構の再設計。§4 の通り費用対効果が成立しない

## 6. まとめ

| 案 | 内容 | FMP11 クラッシュ | FMP11 DDL | 工数 | 判定 |
|---|---|---|---|---|---|
| **A（現状）** | extnVersion ゲート + 呼び出し元 env ドレイン | 解決済み | 不可（仕様化） | 0 | **採用中・維持** |
| B | ラッパー完全互換再実装（FMX 使用） | 変わらない（原因はホスト側） | 変わらない | 中 | 却下（得るものが無い） |
| C | xDBC 経由の新実装 | 回避可 | 可 | 大 | 却下（互換不能・セットアップ地獄） |
| D | プラグイン内 SQL エンジン | — | — | 非現実的 | 論外 |
| **A'** | A + FMP11 で DDL 同期実行を実験→採否 | 解決済み | **可能性あり** | 小 | **次の一手として推奨** |

「新たに生成し直す」戦略はシェル・XML では正解だったが、それは壊れていた層がプラグイン内だったから。SQL は壊れていた層がホスト側であり、既にゲートで解決済み。**再実装は問題解決を早めるどころか、解決済みの問題に工数を投じて何も変えない結果になる。**
